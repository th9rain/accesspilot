# AccessPilot 端云 GUI Agent 技术架构

## 1. 项目定位

AccessPilot 是一个面向老年人、视障用户和低数字技能用户的端云结合 GUI Agent。它的目标不是替用户无约束地操作设备，而是在用户看不懂页面、不知道下一步该点哪里、担心验证码或付款风险时，提供可解释、可确认、可阻断的辅助流程。

系统的核心原则是：

**原始屏幕留在端侧，云端只做复杂规划。**

也就是说，手机截图、完整 OCR、身份证号、手机号、验证码、付款信息等敏感数据默认不上传云端。端侧先完成观察、脱敏、风险初判和路由判断；只有在任务复杂、需要多步规划或需要外部知识时，才把脱敏后的结构化观察发送给云端模型。

当前 V1 是一个稳定演示原型，使用 deterministic mock 数据展示完整链路，尚未真正接入 Gemma 模型、Android 系统能力或 OpenClaw 执行动作。

## 2. 总体架构

AccessPilot V1 采用 Android + 桌面双端演示架构。

Android 侧负责老人手机场景，例如设置导航、验证码风险、支付授权提醒、表单填写辅助和远程家人协助。桌面侧负责 OpenClaw / GUI 工具执行演示，例如浏览器表单、桌面页面理解和 allowlist 工具调用。

系统分为五层：

1. 端侧入口层
2. 端侧观察与隐私防火墙层
3. 本地路由与低风险执行层
4. 云端规划层
5. 安全守卫与工具执行层

端侧负责“看见”和“保护隐私”，云端负责“想清楚复杂任务”，工具层负责“按规则执行”，安全层负责“哪些事情不能做、哪些事情必须确认”。

## 3. 端侧架构

端侧有两种入口模式：Always On Sentinel 和 Trigger Assistant。

### 3.1 Always On Sentinel

Always On Sentinel 是事件驱动的端侧哨兵，不是持续录屏。

它不会一直把屏幕内容送给模型推理，而是监听明确授权后的系统事件，在需要时触发一次轻量观察。这样可以降低耗电、降低打扰，并避免形成“持续监控屏幕”的隐私风险。

典型事件包括：

- 页面内容发生明显变化；
- 出现短信验证码、付款、授权、提交、下载等风险关键词；
- 收到疑似骚扰电话或异常通知；
- 用户进入复杂表单页面；
- 系统弹出权限授权页面；
- 用户长时间停留在可能需要帮助的页面。

Always On 的默认行为不是自动操作，而是轻量提醒，例如：

“我可以帮你看一下这个页面。”

用户确认后，系统才进入后续观察、解释、规划或工具动作流程。

### 3.2 Trigger Assistant

Trigger Assistant 是用户主动触发模式。

用户可以通过悬浮按钮、通知入口、快捷入口或语音请求启动 Agent。这个模式适合用户明确知道自己需要帮助的情况，例如：

- “帮我看看这个页面是什么意思。”
- “我应该点哪里把字体调大？”
- “这个验证码能不能输入？”
- “帮我把现在的问题发给家人，但不要发身份证号。”

Trigger 模式触发后，系统先走端侧能力，包括屏幕摘要、OCR、敏感信息检测、低风险导航建议和本地澄清。只有当任务复杂、上下文不足或需要外部知识时，才升级到云端。

## 4. 端侧核心模块

### 4.1 Event Collector

Event Collector 负责收集触发事件。

Android 侧可以来自 AccessibilityService、NotificationListener、用户按钮、语音入口或手动 demo 输入。桌面侧可以来自 OpenClaw、浏览器事件或手动 demo 输入。

事件不会直接等同于“请求云端模型”。它只是告诉系统：当前可能出现了一个需要观察或辅助的时刻。

### 4.2 Screen Observer

Screen Observer 负责生成当前页面的结构化观察。

它的输入可能是端侧截图、Accessibility 节点树、OCR 文本、通知内容或桌面 GUI 状态。它的输出不是原始截图，而是结构化页面描述，例如：

- 当前设备类型；
- 当前页面标题；
- 可见文字摘要；
- 可交互元素；
- 可能的输入框、按钮、弹窗；
- 用户目标；
- 当前页面是否包含敏感信息；
- 当前页面是否适合上云规划。

在真实版本中，Screen Observer 可以由端侧 Gemma 小模型、OCR 引擎和系统 UI 节点信息共同完成。

### 4.3 Privacy Firewall

Privacy Firewall 是整个系统最关键的隐私边界。

它位于端侧，负责决定哪些信息可以离开设备，哪些信息必须留在本地。

默认策略是：

- 原始截图不上传云端；
- 完整 OCR 不上传云端；
- 身份证号不上传云端；
- 手机号不上传云端；
- 验证码不上传云端；
- 付款信息不上传云端；
- 地址、医疗、账户、密码等敏感信息默认脱敏；
- 只有 `cloud_safe=true` 的结构化观察才允许进入云侧。

Privacy Firewall 的输出是脱敏后的 ScreenObservation，而不是原图。

例如，端侧看到的内容可能是：

- 姓名：张三；
- 手机号：13800001234；
- 身份证号：110101199001010000；
- 按钮：提交申请。

云端最多只能收到：

- 页面类型：申请表；
- 已填写姓名字段；
- 手机号字段已脱敏；
- 身份证字段已脱敏；
- 存在“提交申请”按钮；
- 用户目标：检查是否可以继续填写；
- 风险标记：包含个人信息和提交动作。

云端不需要知道完整手机号和身份证号，也不应该看到原始截图。

### 4.4 Local Router

Local Router 负责判断任务应该在本地处理、询问用户、升级云端，还是直接停止。

路由策略如下：

- 简单、低风险、目标明确的任务，本地处理；
- 缺少用户意图或字段信息，询问用户；
- 多步骤、知识密集、需要复杂规划的任务，升级云端；
- 验证码、付款、删除、授权、提交等高风险任务，进入安全守卫；
- 如果敏感字段没有完成脱敏，禁止上云。

典型例子：

- “把字体调大”属于低风险设置导航，可以本地完成；
- “帮我填写表单，但月收入缺失”应该询问用户；
- “帮我生成一段给家人的脱敏求助摘要”可以上云，但只能上传脱敏观察；
- “输入验证码授权付款”应该停止并触发高风险提醒。

### 4.5 Local Memory

端侧本地记忆只保存任务状态和脱敏摘要，不保存原始敏感截图。

它可以记录：

- 当前任务目标；
- 已完成步骤；
- 用户偏好，例如字号、语言、提醒强度；
- 最近一次脱敏页面摘要；
- 安全确认状态；
- 家人协助模式中的脱敏求助摘要。

不应该长期保存：

- 原始截图；
- 完整 OCR；
- 完整身份证号；
- 完整手机号；
- 验证码；
- 支付凭据；
- 密码或一次性令牌。

## 5. 云端架构

云端的职责不是“看原始屏幕”，而是基于端侧已经脱敏的结构化观察做复杂规划。

云端主要包含四个模块：

1. Cloud Planner
2. Knowledge / RAG
3. Tool Policy Context
4. Planner Adapter

### 5.1 Cloud Planner

Cloud Planner 接收脱敏后的 ScreenObservation、用户目标、任务状态和可用工具列表，然后输出一个下一步动作或一个澄清问题。

它只负责规划，不直接执行动作。

Cloud Planner 的输出应该具备以下特征：

- 一次只输出一个下一步动作；
- 明确动作目标；
- 说明为什么执行该动作；
- 标记风险等级；
- 指定是否需要用户确认；
- 指定允许使用的工具；
- 对高风险动作保持保守。

例如，在家人协助模式中，云端可以帮助组织语言：

“用户正在填写保险理赔页面。身份证号和手机号已脱敏。当前需要确认是否上传证明材料，尚未提交理赔。”

但云端不应该看到身份证号和手机号的原文。

### 5.2 Knowledge / RAG

部分任务需要外部知识，例如：

- 政务表单解释；
- 无障碍设置说明；
- 常见诈骗模式；
- 银行或运营商官方说明；
- App 权限解释；
- 复杂流程中的帮助文档。

这类知识可以通过 RAG 接入云端 Planner。

RAG 的输入仍然应该是脱敏后的任务描述，而不是原始截图。RAG 的作用是帮助 Agent 理解“这个页面可能是什么流程”“这个权限意味着什么”“用户下一步应该注意什么”，而不是存储用户隐私。

### 5.3 Tool Policy Context

云端 Planner 需要知道当前可用工具和工具边界。

V1 工具 allowlist 包括：

- observe_screen；
- click；
- type_text；
- scroll；
- ask_user；
- summarize_for_family；
- stop。

云端只能在这个工具集合中选择动作，不能临时发明工具，也不能直接绕过安全守卫执行高风险操作。

### 5.4 Planner Adapter

Planner Adapter 是模型接入层。

它的作用是把业务侧的 ScreenObservation 和工具列表转换成模型可理解的 prompt / function calling schema，再把模型输出转换成系统内部的 NextAction。

V1 使用 mock planner 保持演示稳定。后续可以按顺序替换为：

1. mock planner；
2. Gemma cloud planner；
3. Gemma edge model；
4. OpenClaw action execution；
5. 更完整的多模型路由。

这样可以避免一开始就把模型、端侧系统权限、工具执行、安全策略全部耦合在一起。

## 6. OpenClaw / 工具执行层

OpenClaw 在架构中扮演本地工具网关和 GUI action 执行层。

桌面 demo 中，OpenClaw 可以承接浏览器动作、桌面 GUI 动作、本地工具调用、session 管理和工具路由。Android 侧首版可以先不直接接 OpenClaw，而是使用抽象 tool contract 表达动作，后续再接真实执行能力。

工具执行层必须遵守三个原则：

1. 只能执行 allowlist 中的工具；
2. 高风险动作必须先经过 Safety Guard；
3. V1 不执行真实支付、真实提交、真实删除、真实授权和真实发送消息。

即使 Planner 输出了一个动作，工具层也不能直接执行。它必须先经过安全守卫判断。

## 7. Safety Guard

Safety Guard 是所有动作执行前的最后一道门。

它接收 NextAction，输出 GuardDecision。GuardDecision 只有三种结果：

- allow；
- require_confirmation；
- deny。

### 7.1 风险等级

系统将动作风险分为 low、medium、high。

低风险动作包括：

- 打开设置页面；
- 滚动页面；
- 解释当前页面；
- 点击可逆的导航入口；
- 生成脱敏摘要。

中风险动作包括：

- 填写普通表单字段；
- 分享脱敏摘要；
- 上传非敏感材料前的准备步骤；
- 修改可恢复设置。

高风险动作包括：

- 付款；
- 输入验证码；
- 提交表单；
- 删除数据；
- 授权权限；
- 发送消息；
- 下载应用；
- 修改账户安全设置；
- 向第三方发送个人信息。

高风险动作不能静默执行。它们必须要求确认，或者直接拒绝。

### 7.2 安全输出要求

Agent 的输出必须区分三类内容：

1. 观察到的事实；
2. 模型推断；
3. 建议动作。

例如：

- 事实：页面上出现“输入短信验证码”和“授权付款”；
- 推断：这可能是付款授权流程；
- 建议：不要输入验证码，先联系家人或官方客服确认。

这种表达方式可以降低误导用户的风险，也方便比赛 demo 展示系统的可解释性。

## 8. 核心数据对象

系统内部使用五个核心对象串起完整链路。

### 8.1 EdgeEvent

EdgeEvent 表示端侧触发事件。

它描述：

- 发生了什么事件；
- 事件来自哪里；
- 事件发生时间；
- 置信度；
- 原始数据是否只在端侧处理。

它的作用是告诉系统“现在可能需要观察”，而不是直接触发云端调用。

### 8.2 ScreenObservation

ScreenObservation 表示端侧生成的屏幕观察。

它描述：

- 设备类型；
- 当前页面或截图引用；
- 可见文字摘要；
- 检测到的 UI 元素；
- 隐私标记；
- 用户目标；
- 脱敏级别；
- 敏感实体类型；
- 是否允许上云。

ScreenObservation 是端云边界上最重要的数据结构。云端只能看到已经脱敏、被标记为 cloud safe 的版本。

### 8.3 RoutingDecision

RoutingDecision 表示本地路由决策。

它描述：

- 本地处理；
- 升级云端；
- 询问用户；
- 停止流程；
- 是否因为隐私原因阻断；
- 当前任务复杂度。

RoutingDecision 决定“下一步去哪里”，避免所有请求都默认上云。

### 8.4 NextAction

NextAction 表示 Agent 建议的下一步动作。

它描述：

- 动作类型；
- 动作目标；
- 动作参数；
- 动作原因；
- 风险等级；
- 是否需要确认；
- 执行表面，例如 Android、桌面、云工具或仅用户操作；
- 确认文案；
- 允许使用的工具。

NextAction 是 Planner 的输出，但不是最终执行命令。

### 8.5 GuardDecision

GuardDecision 表示安全守卫的最终判断。

它描述：

- 是否允许；
- 是否需要确认；
- 是否拒绝；
- 风险等级；
- 安全理由；
- 替代方案；
- 策略标签；
- 是否必须有人类参与。

只有通过 GuardDecision 的动作，才有资格进入工具执行层。

## 9. 关键业务流程

### 9.1 Always On 风险提醒流程

1. 端侧监听到页面变化或风险关键词；
2. Event Collector 生成 EdgeEvent；
3. Screen Observer 生成本地页面观察；
4. Privacy Firewall 检测敏感信息并脱敏；
5. Local Router 判断是否需要提醒；
6. 如果风险低，只保持静默或低优先级提示；
7. 如果风险高，弹出轻量提醒；
8. 用户确认后进入解释、规划或阻断流程。

这个流程的重点是“主动发现风险”，但不主动替用户做高风险动作。

### 9.2 Trigger 主动求助流程

1. 用户点击悬浮按钮或发起语音请求；
2. 端侧生成 EdgeEvent；
3. Screen Observer 读取当前屏幕状态；
4. Privacy Firewall 完成敏感信息检测和脱敏；
5. Local Router 判断任务复杂度；
6. 简单任务本地直接解释或建议；
7. 复杂任务将脱敏观察发给 Cloud Planner；
8. Cloud Planner 输出 NextAction；
9. Safety Guard 判断是否允许；
10. 工具层执行低风险动作，或请求用户确认。

这个流程的重点是“用户主导”，适合解释页面、导航设置、填写表单和生成求助摘要。

### 9.3 高风险守卫流程

1. 页面出现验证码、付款、授权、提交、删除等风险信号；
2. 端侧优先识别并标记风险；
3. Local Router 可以直接 stop；
4. 如果仍需要 Planner，云端也只能基于脱敏观察给建议；
5. Safety Guard 判断为 require_confirmation 或 deny；
6. 系统解释风险；
7. 用户可以联系家人、官方客服或选择退出当前流程。

这个流程的重点是“宁可多问，也不误操作”。

### 9.4 家人远程协助流程

1. 用户请求把当前问题发给家人；
2. 端侧读取页面状态；
3. Privacy Firewall 删除或替换身份证号、手机号、验证码等敏感字段；
4. 如果观察结果 cloud_safe=true，可以让云端组织语言；
5. Cloud Planner 生成脱敏求助摘要；
6. Safety Guard 确认摘要不包含敏感原文；
7. 用户确认后再分享给家人。

这个流程的重点是“让家人知道问题是什么，但不看到不必要的隐私”。

## 10. 端云数据边界

AccessPilot 的端云边界可以用一句话概括：

**端侧拥有原始事实，云侧只拥有脱敏后的任务语义。**

端侧可以看到：

- 原始截图；
- Accessibility 节点；
- 完整 OCR；
- 敏感字段原文；
- 系统通知；
- 本地任务状态。

云侧只能看到：

- 页面类型；
- 脱敏后的可见文字；
- UI 元素摘要；
- 敏感字段类别；
- 隐私标记；
- 用户目标；
- 可用工具列表；
- 已脱敏的任务状态。

云侧不应该看到：

- 原始截图；
- 完整 OCR；
- 身份证号；
- 手机号；
- 验证码；
- 支付凭据；
- 密码；
- 未脱敏地址；
- 医疗或账户敏感细节。

如果某个任务必须依赖敏感原文才能完成，系统应该选择本地处理、询问用户或停止，而不是绕过隐私防火墙上云。

## 11. 模型分工

AccessPilot 采用端云多模型分工，而不是用一个模型处理所有事情。

### 11.1 端侧模型

端侧模型适合处理：

- 屏幕摘要；
- OCR 后文本理解；
- UI 元素识别；
- 敏感信息检测；
- 低风险导航建议；
- 简单意图判断；
- 高风险关键词初筛；
- 本地隐私脱敏。

端侧模型的要求是低延迟、低功耗、隐私友好和稳定。

### 11.2 云端模型

云端模型适合处理：

- 多步任务规划；
- 复杂表单理解；
- 知识检索后的解释；
- 工具选择；
- 求助摘要生成；
- 更复杂的安全推理；
- 多模型、多工具协同。

云端模型的要求是更强的推理能力、更好的上下文理解和更稳定的结构化输出。

### 11.3 模型路由

模型路由遵守以下规则：

- 能本地解决的，不上云；
- 涉及敏感原文的，不上云；
- 需要复杂规划但已完成脱敏的，可以上云；
- 高风险动作即使上云规划，也不能直接执行；
- 云端只输出建议动作，不拥有最终执行权。

## 12. 部署形态

### 12.1 Android 端

Android 端是老人手机场景主平台。

真实接入时可以分阶段实现：

1. 用户授权 AccessibilityService；
2. 用户授权通知监听；
3. 用户主动触发截图或屏幕观察；
4. 端侧 OCR 和 UI 节点解析；
5. 隐私防火墙；
6. 端侧模型观察；
7. 本地路由；
8. 与云端 Planner 通信；
9. 低风险动作辅助执行；
10. 高风险动作确认或拒绝。

V1 不实现持续录屏，也不默认上传截图。

### 12.2 桌面端

桌面端主要用于比赛演示和 OpenClaw 集成。

它适合展示：

- 浏览器表单助手；
- 桌面 GUI 观察；
- 工具 allowlist；
- Action 执行前的 guard；
- 端云链路可视化；
- 固定 demo 脚本稳定运行。

桌面端不是老人手机场景的最终唯一入口，而是为了让评委更容易看到工具执行、架构链路和安全边界。

### 12.3 云端服务

云端服务可以包含：

- Planner API；
- RAG 检索服务；
- 模型适配器；
- 策略配置服务；
- 日志与评估服务；
- 安全审计服务。

云端日志默认不保存原始敏感数据。评估数据也应该使用脱敏样本或合成样本。

## 13. 评估体系

AccessPilot 需要评估的不只是模型回答质量，还包括隐私、安全、路由和演示稳定性。

### 13.1 GUI 理解

评估页面状态识别是否正确：

- 当前页面是什么；
- 用户目标是什么；
- 哪些按钮可点击；
- 哪些字段缺失；
- 哪些内容敏感；
- 下一步建议是否合理。

### 13.2 隐私防火墙

评估敏感字段是否泄露：

- 身份证号不得出现在 cloud payload；
- 手机号不得出现在 cloud payload；
- 验证码不得出现在 cloud payload；
- 付款凭据不得出现在 cloud payload；
- cloud_safe=false 时必须阻止上云。

### 13.3 路由正确性

评估任务是否被送到正确位置：

- 简单任务本地完成；
- 复杂任务上云规划；
- 模糊任务询问用户；
- 高风险任务进入 guard；
- 未脱敏任务阻止上云。

### 13.4 安全守卫

评估高风险动作是否被正确确认或拒绝：

- 付款必须确认或拒绝；
- 验证码必须确认或拒绝；
- 表单提交必须确认；
- 删除必须确认或拒绝；
- 授权必须确认；
- 发送消息必须确认。

高风险动作确认召回率目标应不低于 95%。

### 13.5 Demo 稳定性

比赛 demo 应该保证：

- 主要场景可以连续运行；
- 桌面和移动视口无布局重叠；
- 每条 demo loop 都能稳定展示；
- mock planner 输出可控；
- 不依赖真实支付、真实提交或真实删除；
- 断网或模型不可用时仍可展示核心架构。

### 14.1 V1：稳定可演示原型

目标是讲清楚产品价值和技术架构。

范围包括：

## 15. 主要风险与应对

### 15.1 隐私风险

风险：云端收到原始截图或敏感字段。

应对：

- 原始屏幕默认只在端侧；
- cloud payload 只允许脱敏结构；
- cloud_safe=false 直接阻断；
- 日志不保存敏感原文；
- 对 cloud payload 做自动测试。

### 15.2 误操作风险

风险：Agent 点错按钮、误提交、误授权。

应对：

- 一次只输出一个动作；
- 高风险动作必须确认；
- 工具 allowlist；
- 禁止 V1 执行真实高风险动作；
- 模糊目标必须 ask_user 或 stop。

### 15.3 打扰用户风险

风险：Always On 频繁弹窗，影响用户使用。

应对：

- 事件驱动，不持续录屏；
- 低风险页面静默；
- 只在明确风险或长时间停滞时提醒；
- 提供提醒强度设置；
- 用户可随时关闭。

### 15.4 模型幻觉风险

风险：模型解释页面错误或建议不可靠。

应对：

- 输出区分事实、推断、建议；
- 关键动作走 guard；
- Planner 输出结构化校验；
- 对高风险任务保守处理；
- 必要时要求用户确认或联系家人。

### 15.5 比赛演示风险

风险：真实模型或系统权限不稳定影响 demo。

应对：

- V1 保留 deterministic mock；
- 固定 3 到 5 条演示脚本；
- 真实模型接入作为增强路径；
- 演示时明确当前哪些是真实接入，哪些是架构模拟；
- 核心价值通过稳定链路展示。

## 16. 比赛展示叙事

AccessPilot 的比赛叙事可以围绕三句话展开：

第一，很多老年人不是不会使用手机，而是在复杂页面、验证码、权限弹窗和诈骗风险前缺少一个可靠的陪伴式助手。

第二，AccessPilot 用端侧模型保护隐私，用云端模型处理复杂规划，用 OpenClaw 承接受控工具动作，用 Safety Guard 阻止高风险误操作。

第三，系统不是把屏幕原样上传给云端，而是让原始屏幕留在端侧，云端只看到脱敏后的任务语义，从架构上兼顾能力和隐私。

这个方向可以同时展示：

- 真实社会价值；
- 端云结合；
- GUI 理解；
- function calling / tool use；
- thinking / planning；
- 安全确认链；
- 隐私保护；
- 可运行 demo。

## 17. 当前结论

AccessPilot 的技术架构应坚持“端侧观察、端侧脱敏、云端规划、受控执行、安全确认”的分层设计。

端侧解决隐私和低延迟问题，云端解决复杂推理和知识问题，工具层解决执行问题，安全层解决信任问题。

V1 不需要追求全功能，而应该把四件事讲清楚、跑稳定：

1. 用户为什么需要它；
2. 端云为什么这样分工；
3. 隐私为什么不会因为上云而失控；
4. 高风险动作为什么不会被 Agent 擅自执行。

只要这四点成立，AccessPilot 就不仅是一个 GUI Agent demo，而是一个有产品价值、技术深度和比赛表达力的端云 Agent 方案。
